# Лабораторная работа №5
## Оптимизация запросов с помощью индексов и анализа плана выполнения.

**Номер варианта:** 13

---

# Цель работы

Изучить принципы работы планировщика запросов PostgreSQL, освоить команды `EXPLAIN` и `EXPLAIN ANALYZE` для диагностики производительности, научиться создавать и применять индексы для ускорения выборок данных. Научиться анализировать производительность SQL-запросов, интерпретировать план выполнения (Query Plan) и оптимизировать работу базы данных с помощью различных типов индексов (B-tree, Hash).

---

# Задачи

- Изучить методы сканирования таблиц: последовательное сканирование (Sequential Scan) и сканирование по индексу (Index Scan / Bitmap Heap Scan)
- Освоить команды `EXPLAIN` и `EXPLAIN ANALYZE` для анализа плана выполнения запросов
- Научиться создавать индексы типа B-Tree для оптимизации поиска по условиям равенства
- Сравнить производительность запросов до и после создания индексов

## Задание 3
Оптимизировать `JOIN sales` и `customers` по `customer_id`.

[Скриншот запроса](https://raw.github.com/MakhoninaAV/sql_files/main/%D0%BB%D0%B0%D0%B1%205%20%D1%84%D0%BE%D1%82%D0%BE/lab_05%20%D0%B7%D0%B0%D0%B4%203.1%20%D0%B0%D0%BD%D0%B0%D0%BB%D0%B8%D0%B7%20%D0%B4%D0%BE%20%D0%B8%D0%BD%D0%B4%D0%B5%D0%BA%D1%81%D0%B0.png)

Результат запроса `EXPLAIN ANALYZE` до создания индекса

- **Метод соединения:** Hash Join

- **Метод сканирования:** Sequential Scan (последовательное сканирование) на обеих таблицах

- **Строки:** найдено 37711 записей в таблице sales, 50000 записей в таблице customers

- **Время выполнения:** 33.677 мс

Вывод: PostgreSQL выполняет полное сканирование обеих таблиц. Таблица `sales` (37711 строк) читается целиком, затем строится хеш-таблица из `customers` (50000 строк), после чего происходит соединение. Это занимает значительное время, особенно на больших объемах данных.

[Скриншот запроса](https://raw.github.com/MakhoninaAV/sql_files/main/%D0%BB%D0%B0%D0%B1%205%20%D1%84%D0%BE%D1%82%D0%BE/lab_05%20%D0%B7%D0%B0%D0%B4%203.2%20%D0%B0%D0%BD%D0%B0%D0%BB%D0%B8%D0%B7%20%D0%BF%D0%BE%D1%81%D0%BB%D0%B5%20%D0%B8%D0%BD%D0%B4%D0%B5%D0%BA%D1%81%D0%B0.png)

Результат запроса EXPLAIN ANALYZE после создания индекса

- **Метод сканирования:** Sequential Scan (последовательное сканирование) — индекс не используется

- **Index Cond:** индекс `idx_sales_customer_id` создан, но не применяется планировщиком

- **Строки:** обработано 37711 строк в таблице sales, 50000 строк в таблице customers

- **Время выполнения:** 36.136 мс

**Сравнение производительности:**
- До создания индекса использовалось последовательное сканирование (Seq Scan) с временем выполнения 33.677 мс. Было обработано 37711 строк из sales и 50000 строк из customers.
- После создания индекса PostgreSQL продолжает использовать Sequential Scan. Время выполнения незначительно увеличилось до 36.136 мс. Индекс `idx_sales_customer_id` не был использован, так как планировщик оценил, что для полного JOIN-соединения (без фильтрации) хеш-соединение с последовательным сканированием обеих таблиц будет эффективнее, чем использование индекса.

**Вывод:**
Индекс на поле customer_id в таблице sales не дал выигрыша в производительности для данного запроса, так как запрос выбирает все строки из обеих таблиц (37711 из sales и 50000 из customers). При выборке большого процента строк (в данном случае 100%) последовательное сканирование и хеш-соединение работают быстрее, чем индексное сканирование с последующим доступом к таблице. Индексы эффективны только при выборке малого процента строк (обычно менее 5-10%).

# Вывод

В ходе выполнения лабораторной работы были изучены методы оптимизации запросов с использованием индексов в PostgreSQL. С помощью команды `EXPLAIN ANALYZE` проведен анализ планов выполнения запросов, что позволило сравнить производительность до и после создания индексов.

**Основные выводы по работе:**

1. Индексы эффективны только при выборке **малого процента строк** (обычно менее 5-10%). При выборке всех строк (100%) последовательное сканирование работает быстрее.
2. На маленьких таблицах (менее нескольких тысяч строк) индекс может не использоваться, даже если он создан — планировщик выбирает более дешевый план.
3. Команда `EXPLAIN ANALYZE` позволяет точно определить, используется ли индекс и насколько он улучшает производительность.
4. Время планирования (Planning Time) может увеличиваться при наличии индексов, так как оптимизатор рассматривает больше вариантов планов выполнения.

Таким образом, индексы — мощный инструмент оптимизации, но их создание должно быть обоснованным. Необходимо анализировать план выполнения запроса с помощью `EXPLAIN ANALYZE`, чтобы убедиться, что индекс действительно используется и дает выигрыш в производительности.
